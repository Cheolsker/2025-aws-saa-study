# 3번
## 정답
- D

## 근거
- A/B ❌
  - A/B 옵션처럼 Lambda 함수를 데이터베이스 업데이트 시점에 직접 생성하는 것은 데이터베이스를 모니터링하고 알림을 보내는 데 추가적인 복잡성을 야기하며, RDS의 내장 기능(이벤트 알림)을 활용하지 않아 비효율적
  
- C ❌
  - RDS 이벤트 알림 -> SQS -> SNS로 분산
  - SQS가 다시 SNS로 분산하는 것은 표준적인 아키텍쳐가 아님 (SNS -> 여러 SQS가 일반적)

- D ✅
  - `RDS 이벤트 알림 -> SNS 주제 -> 여러 SQS 대기열 -> Lambda (대상 시스템 업데이트)`
  - `AWS SNS (Simple Notification Service)`
    - 하나의 메시지를 여러 구독자에게 브로드캐스트 (여러 대상 시스템)
  - `AWS SQS (Simple Queue Service)`
    - SQS는 메시지를 안정적으로 저장하고, 메시지를 손실 없이 소비자(Consumer)에게 전달하며, 소비자(대상 시스템)의 부하에 맞춰 비동기적으로 처리
  - `AWS Lambda`
    - SQS 대기열의 메시지를 가져와(Poll) 각 대상 시스템의 API를 호출하거나 필요한 방식으로 데이터를 처리하고 업데이트하는 데 사용
    - Lambda는 서버리스 방식으로 확장성이 뛰어나고 SQS를 트리거로 사용하기에 적합합니다.



# 4번
## 정답
- B

## 근거
- `EC2 구매옵션`
  - 예약 인스턴스
    - 1년 또는 3년 약정을 통해 **온디맨드 요금 대비 최대 72%**까지 할인
    - 실행 시간이 길고 중단되어서는 안되는 워크로드에 효율적
  - 온디맨드
    - 사용한 만큼만 비용을 지불
    - 인스턴스가 중단되면 비용이 발생하지 않습니다. 
    - 실행 시간이 짧고 불규칙하며, 사용하지 않을 때 중지할 수 있음
  - 스팟 인스턴스
    - 언제든지 중단될 수 있음
    - 중단되어서는 안되는 워크로드에 부적합

- 결론
  - 프로덕션 (24시간 실행, 중단 불가)
    -> **예약 인스턴스 (Reserved Instances)**로 가장 큰 폭의 할인
  - 개발/테스트 (8시간 실행, 중단 가능)
    -> **온디맨드 인스턴스 (On-Demand Instances)**로 유연성을 확보하고, 중단 시 즉시 비용을 절감
