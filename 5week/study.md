- 매일 30TB가 넘는 클릭스트림 데이터 생성
- 클릭스트림: 사용자가 웹사이트나 애플리케이션을 이용하는 과정에서 발생하는 모든 행동(클릭, 페이지 이동, 스크롤, 입력 등) 데이터를 시간 순서대로 기록한 것

예상정답 > D
- 모르는 키워드들이 많은데, 분석을 위한 AWS 서비스를 이용해야 할 것 같다.
<br/>
정답 > D

## 근거
- Amazon Kinesis Data Streams (KDS)
  - 수백만 대의 소스에서 페타바이트 규모의 데이터를 안정적으로 수집하고 실시간으로 보존하는 핵심 계층
  - 글로벌 사이트에서 발생하는 대용량 스트리밍 데이터를 실시간으로 수집하고 버퍼링할 수 있음
- Amazon Kinesis Data Firehose
  - 스트리밍 데이터를 지정된 저장소(S3, Redshift 등)로 자동으로 로드하는 완전 관리형 서비스.
  - 데이터를 S3 데이터 레이크로 전송하는 작업을 간편하고 안정적으로 수행, 이후 Amazon Redshift로 로드하는 파이프라인을 쉽게 구축가능
- Amazon Redshift
  - 대규모 데이터 웨어하우스(Data Warehouse) 서비스
  - 수집된 데이터를 사용하여 분석을 생성하고, 대량의 데이터에 대한 복잡한 쿼리를 빠르게 처리가능.

---

- A
  - AWS Data Pipeline은 데이터 전송을 자동화하는 데 사용되지만, 실시간 스트리밍 데이터 처리보다는 **배치(Batch)** 또는 주기적인 데이터 이동에 더 적합 
  - 실시간/준실시간 요구 사항에 맞춰 수백 개의 소스에서 발생하는 대량 클릭스트림을 수집하기 어려움.
- B
  - EC2 인스턴스를 사용하여 실시간 데이터 수집 및 전송 파이프라인을 직접 구축하는 것은 가능
  - 그렇지만 **운영 부담(관리 오버헤드)**이 매우 높고, Kinesis 서비스와 비교했을 때, 확장성 및 안정성을 확보하는 것이 비효율적
- C Amazon CloudFront
  - **CloudFront**는 콘텐츠 전송 네트워크(CDN), **정적 콘텐츠**를 캐시하여 사용자에게 빠르게 전달하는 서비스
  - 클릭스트림 데이터 자체를 분석을 위해 수집하고 처리하는 플랫폼 역할은 할 수 없음.
  - AWS Lambda는 데이터 처리에는 유용하지만, 대용량 스트림 데이터를 수집하는 역할은 Kinesis가 더 적합


## 사전지식
- S3 데이터 레이크
  <details>
    <summary>내용토글</summary>
    <br/>

    S3 데이터 레이크는 무제한의 저렴한 스토리지와 데이터 처리의 유연성을 결합한 클라우드 아키텍처의 기반입니다.

    1) 무제한의 확장성과 비용 효율성 💰
    ```
    무제한 확장성 (Massive Scalability): 사실상 저장할 수 있는 데이터의 양에 제한이 없습니다. 페타바이트(PB) 이상의 데이터도 쉽게 수용하며, 자동으로 스토리지 크기를 확장합니다.

    저렴한 비용 (Low Cost): 객체 스토리지를 기반으로 하므로, 블록 스토리지나 데이터베이스보다 훨씬 저렴한 비용으로 대규모 데이터를 저장할 수 있습니다.

    컴퓨팅-스토리지 분리 (Decoupled Architecture): 데이터 저장(S3)과 데이터 분석(Athena, EMR 등)을 위한 컴퓨팅 리소스가 분리되어 있어, 스토리지 용량이 증가해도 분석 비용에 영향을 주지 않아 비용 효율성이 높습니다.
    ```

    2) 데이터 형식의 유연성 ✨
    ```
    원시 데이터 저장 (Raw Data Storage): 데이터를 정제하거나 구조화하지 않고, 원본 형태(Raw Data) 그대로 저장합니다.

    모든 데이터 유형 지원: 정형 (CSV, Parquet), 반정형 (JSON, XML, 로그), 비정형 (이미지, 비디오) 데이터를 모두 단일 저장소에 통합합니다.

    읽기 시 스키마 정의 (Schema-on-Read): 데이터를 저장할 때 스키마를 미리 정의할 필요 없이, 분석을 수행하는 시점에 필요한 스키마를 적용합니다. 이 유연성은 탐색적 분석이나 **머신러닝(ML)**에 매우 유리합니다.
    ```

    3) 강력한 안정성 및 보안 🛡️
    ```
    최고 수준의 내구성 (Durability): 99.999999999% (11-9s)의 내구성을 제공하여 데이터 손실 위험을 거의 없앱니다.

    고가용성 (High Availability): 여러 가용성 영역(AZ)에 데이터를 중복 저장하여 높은 가용성을 보장합니다.

    중앙 집중식 거버넌스: AWS Lake Formation과 같은 서비스를 사용하여 S3 내의 데이터에 대한 접근 권한, 보안 정책, 감사를 중앙에서 세밀하게 관리할 수 있습니다.
    ```

    4) 분석 생태계의 허브 역할 🔗
    ```
    다양한 도구 통합: AWS의 모든 주요 분석 서비스(Amazon Athena, Amazon Redshift Spectrum, AWS Glue, Amazon EMR, Amazon SageMaker)가 S3에 직접 연결되어 데이터를 복사할 필요 없이 쿼리 및 분석을 수행할 수 있습니다.

    데이터 파이프라인의 핵심: 실시간 스트리밍 데이터(Kinesis), 배치 데이터(AWS DataSync), 데이터베이스 복제 데이터 등 모든 데이터 파이프라인의 최종 목적지이자 중앙 허브 역할을 수행합니다.
    ```
  </details>

  - 처리된 데이터는 저지연 검색을 위한 NoSQL 데이터베이스 DynamoDB에 효율적으로 저장될 수 있다.

- A
  - DynamoDB Streams는 데이터가 DynamoDB에 쓰여진 후 변경사항을 캡처

- B

  - 다른 애플리케이션이 S3에서 데이터를 사용하는 것은 실시간으로 보기 어려움
  - S3에 데이터를 저장해서 정보를 요청하는 것은 저지연 검색 요구사항을 충족시키지 못함
- D
  - 일괄처리 방식으로 S3에 저장하고 주기적으로 Lambda가 처리하는 것은 실시간 처리 요구사항과 거리가 멈.

## 근거
- SQS
  - API Gateway를 사용하지 않는다면, 수백만 개의 장치에서 들어오는 요청을 비동기적으로 수용하여 버퍼링하고 부하를 분산하는데 효과적
- EC2 인스턴스를 이용한 처리
  - SQS의 무한한 확장성 덕분에, EC2 인스턴스는 들어오는 메시지 속도에 맞춰 탄력적으로 처리가능
- API Gateway (수집 계층)
  - 수백만~수천만 건의 요청을 확장 가능하게 수용하고 관리하는데 적합한 완전 관리형 서비스
  - EC2 인스턴스를 직접 노출하는 것보다 운영 오버헤드가 낮다.
- Kinesis Data Streams/Firebose (전송 및 저장)
  - API Gateway와 통합하여 데이터를 거의 실시간으로 수집 가능
  - 데이터 변환 (Lambda 통합) 및 여러 애플리케이션의 데이터 공유가 쉽다
  - Firehose의 내장된 변환 기능을 사용하여 데이터를 변환하고 S3에 저장하여 기존 EC2의 데이터 변환 및 저장 역할 대체할 수 있다.

- A
  - Glue는 S3에 이미 저장된 데이터(Data Lake)를 대상으로 하는 일괄처리 서비스
  - 실시간 수집 파이프라인의 첫번째 단계로 적합하지 않음
  - 현재 EC2가 처리하는 변환작업을 대체할 수 없음
- B
  - Route 53은 DNS 서비스로, 트래픽 라우팅은 돕지만 EC2 자체의 확장성 문제나 운영오버헤드 증가 문제를 직접적으로 해결 못함
  - EC2 인스턴스를 계속 사용하면 오토스케일링 그룹과 로드밸런서를 계속 관리해야 함.
- C
  - 확장성 문제를 부분적으로 해결하지만 EC2 인스턴스 증가는 운영 오버헤드 (패치, 관리, 모니터링 등..)를 증가시킴
  - API Gateway, Kinesis를 사용하는 것이 더 나은 솔루션.

## 근거
- AWS Global Accelerator
  - AWS 글로벌 네트워크를 활용하여 사용자의 애플리케이션에 대한 트래픽 성능 향상 & 지연 시간을 줄이는 서비스
    - 사용자로부터 가장 가까운 엣지 로케이션(Edge Location)을 통해 트래픽을 수신한 후, AWS의 전용 글로벌 백본 네트워크를 통해 가장 적절한 AWS 리전의 엔드포인트(여기서는 NLB)로 전송
    - 일반 인터넷을 사용하는 것보다 네트워크 혼잡과 홉(hop) 수를 줄여 전송 속도와 일관성을 높입니다.
  - TCP 및 UDP 트래픽 지원
  - Global Accelerator를 사용하려면 애플리케이션이 사용하는 포트(게임의 TCP/UDP 포트)에 맞게 리스너를 구성하고 해당 포트를 NLB 엔드포인트로 연결해야 함.

- A
  - CloudFront는 주로 HTTP/HTTPS 기반의 정적 및 동적 콘텐츠 캐싱, 배포를 위한 CDN(Content Delivery Network)
  - 캐싱을 늘리는 것은 지연 시간이 중요한 라이브 게임 플레이에는 도움이 되지 않음
- B
  - ALB는 레이어 7(HTTP/HTTPS) 로드 밸런서이므로 UDP 트래픽을 지원하지 않음
  - 대기 시간 기반 라우팅은 사용자 요청을 대기 시간이 가장 짧은 리전으로 보내지만, 그 이후의 트래픽은 여전히 공용 인터넷을 통과
  - Global Accelerator는 AWS 전용 백본 네트워크를 사용하여 공용 인터넷의 변동성을 피하므로, 성능 개선 및 지연 시간 감소 측면에서 Route 53의 대기 시간 기반 라우팅보다 더 우수한 솔루션
- D
  - API Gateway는 일반적으로 RESTful API, WebSocket API 또는 HTTP API를 구축하는 데 사용
  - 캐싱을 활성화하는 것은 게임 플레이와 같은 실시간 상호 작용에는 부적합

## 키워드
- 대량의 데이터를 저장한다.
- 매시간 데이터를 분석, 여러 가용성영역(AZ) 에 배포

예상정답 > C
- EFS는 여러 AZ에 파일을 관리/대응할 수 있고, 유동적으로 증가하는 파일을 관리할 수 있음, 매시간 파일액세스 허용
- S3 Glancier는 장기적으로 파일을 관리하는데 유리, 파일액세스가 수~수십분 걸리므로 오답
- EBS...
- EBS은 그렇다치고 프로비저닝한다는 것이 문제, 여기서는 6개월간 파일의 저장공간이 증가한다고 했음, 프로비저닝한 저장공간보다 파일용량이 크면 어쩔?
<br/>
정답 > C

## 근거
- EFS
  - EFS는 파일 시스템 크기를 자동으로 확장/축소하며, 페타바이트 규모까지 확장 가능한 탄력적인 스토리지
  - EFS는 **NFSv4 프로토콜**을 사용하여 **여러 가용성 영역(Multi-AZ)**에 걸쳐 배포된 수많은 Amazon EC2 인스턴스가 **동시에** 파일 시스템을 마운트하고 데이터를 읽고 쓸 수 있도록 지원함.
  - EFS는 데이터에 대한 높은 처리량 및 낮은 지연 시간 액세스를 제공하여 빈번한 분석 및 수정 작업에 적합 (매시간 분석가능)
| - EFS는 여러 AZ에 걸쳐 데이터를 복제하여 고가용성을 제공하며, 다른 AZ의 인스턴스에서도 파일 시스템을 마운트할 수 있음

---

- A
  - S3 Glacier는 주로 **장기 보관 및 아카이빙**을 위한 서비스
  - 매시간 데이터를 수정하고 분석하는 **빈번한 액세스** 워크로드에는 적합하지 않다.
- B
  - EBS 볼륨은 한 번에 **하나의 EC2 인스턴스에만** 마운트될 수 있음
  - 문제에서는 **"여러 Amazon EC2 Linux 인스턴스"**가 데이터를 수정해야 하므로 부적합
