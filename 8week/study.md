## 근거
- A ❌
  - Fargate는 컨테이너화된 애플리케이션을 실행하기 위한 서버리스 컴퓨팅 서비스
  - 정적 콘텐츠 제공을 위해 컨테이너와 컴퓨팅 자원을 사용하는 것은 과도한 리소스 낭비
  - S3에 비해 훨씬 비용이 많이 듭니다.

- B ✅
  - S3는 본래 객체(파일) 저장 서비스이지만, 정적 웹사이트 호스팅 기능을 기본적으로 제공
  - 웹 서버(EC2, Fargate 등)를 상시 운영할 필요 없이, 실제로 사용된 스토리지 용량, 데이터 전송량 및 요청 수에 대해서만 비용을 지불 
  - 트래픽이 적은 경우 다른 옵션에 비해 압도적으로 비용이 저렴하며, 사실상 웹 서버 운영 비용이 들지 않는 경우가 많음.

- C ❌
  - EC2 인스턴스는 24시간 실행되어야 하므로, 해당 인스턴스 비용(시간당)이 발생
  - 정적 웹사이트의 경우 EC2 인스턴스를 유지 관리하는 것은 S3 호스팅에 비해 비용 효율성이 떨어지고 관리 부담이 크다.

- D 
  - Lambda는 서버리스 컴퓨팅 서비스이지만, 코드를 실행하여 동적 콘텐츠를 생성하는 데 주로 사용 
  - 정적 콘텐츠를 제공하기 위해 복잡한 Express.js 애플리케이션과 로드 밸런서를 구성하는 것은 불필요하게 복잡
  - S3에 비해 요청 처리 비용이 더 발생할 수 있음, Lambda는 정적 파일 제공의 주된 목적이 아님.



# 8번
## 정답
- B

## 근거
- A ❌
  - 프라이빗 서브넷의 경로 테이블에 IGW를 직접 연결하면 해당 서브넷은 퍼블릭 서브넷이 되어 버림
  - 인스턴스가 직접 인터넷에 노출되어 보안 요구 사항을 위반

  - `인터넷 게이트웨이`
    - 인터넷 연결을 담당
    - 인바운드/아웃바운드 트래픽 허용
    - 공인 IP주소를 사설 IP로 변환하여 해당 EC2 인스턴스에 요청 보냄

- B ✅
  - `프라이빗 서브넷`
    - **인터넷 게이트웨이(Internet Gateway, IGW)**로 향하는 직접적인 경로가 없는 서브넷
    - 인스턴스는 외부에서 들어오는 트래픽에 노출되지 않으며, 아웃바운드 인터넷 액세스도 기본적으로 불가능 (보안이 강화됨)
  
  - `NAT (Network Address Translation) 게이트웨이`
    - 프라이빗 서브넷의 인스턴스가 인터넷에 아웃바운드 트래픽을 보낼 수 있게 해주는 AWS 관리형 서비스
    - 인바운드 트래픽 허용 X, 아웃바운드 트래픽 허용 O
    - NAT 게이트웨이는 퍼블릭 서브넷에 배치되어야 하며, 퍼블릭 서브넷은 인터넷 게이트웨이에 연결되어 있어야 함

  - `NAT 구성방법`
    - NAT 게이트웨이를 퍼블릭 서브넷에 생성
    - EC2 인스턴스가 속한 프라이빗 서브넷의 경로 테이블을 수정.
    - 기본 경로(`Destination: 0.0.0.0/0`)의 대상으로 방금 생성한 NAT 게이트웨이를 지정
    - 이렇게 하면 프라이빗 인스턴스의 모든 외부 트래픽이 NAT 게이트웨이를 통해 인터넷 게이트웨이로 전달가능

- C ❌
  - `NAT 인스턴스`
    - NAT 게이트웨이의 이전 버전/대체 솔루션
    - NAT 인스턴스 자체는 퍼블릭 서브넷에 배치되어야 인터넷과 통신할 수 있음
    - 프라이빗 서브넷에 배치하면 아웃바운드 통신이 불가능

- D ❌
  - 기본 경로를 IGW로 직접 지정하는 것은 A와 동일한 문제(프라이빗 서브넷이 퍼블릭 서브넷이 됨)
  - NAT 인스턴스를 사용하더라도 경로를 NAT 인스턴스로 지정해야 한다.
  - 또한 NAT 인스턴스는 프라이빗 서브넷이 아닌 퍼블릭 서브넷에 배치되어야 한다.



# 9번
## 요구사항
- 애플리케이션의 성능 문제가 데이터베이스 쿼리 속도 
- **읽기 부하(Read Traffic)**가 주 인스턴스에 집중되어 발생했음을 의미

## 정답
- C

## 근거
- A ❌
  - Multi-AZ 복제본은 **재해 복구(Disaster Recovery) 및 고가용성(High Availability)**을 위해 사용
  - 이 대기 복제본은 평소에는 액세스할 수 없으며 오직 장애 발생 시에만 활성화됨
  - 따라서 성능 개선을 위한 읽기 부하 분산에 사용할 수 없음. (RDS Multi-AZ와 Read Replica는 목적이 다릅니다.)

- B ❌
  - `전송 가속(Transfer Acceleration)`
    - AWS S3 서비스의 기능
    - 원거리에서 데이터를 빠르게 전송하기 위해 사용, RDS와는 관련이 없는 솔루션입니다.

- C ✅
  - `읽기 복제본 (Read Replica)`
    - Amazon RDS의 읽기 복제본은 원본(소스) DB 인스턴스의 데이터 사본을 생성하여 읽기 전용 트래픽을 처리하도록 설계된 기능
    - 애플리케이션의 읽기 쿼리를 읽기 복제본으로 분산시키면, 원본 DB 인스턴스는 쓰기 작업(Write Traffic)에만 집중할 수 있게 되어 전체적인 데이터베이스 성능이 향상됨.
    - 읽기 부하로 인한 성능 문제를 해결하는 가장 표준적이고 비용 효율적인 방법입니다.

- D ❌
  - `Kinesis Data Firehose`
    - 스트리밍 데이터를 수집하여 S3, Redshift 등의 대상으로 로드하는 서비스 
    - 대량의 실시간 로그 또는 이벤트 데이터를 처리하는 데 사용
    - 일반적인 애플리케이션의 느린 SQL 쿼리 성능 문제를 해결하는 데는 적합하지 않음



# 13번
## 요구사항
- 안전한 인증: 보안을 최우선
- 정적 자격 증명 회피: 사용자 이름/비밀번호를 관리하거나 코드에 포함하는 것을 원하지 않음
- 최소 운영 노력: AWS 관리형 기능을 사용하여 인프라 관리 부담을 줄여야

## 정답
- C

## 근거
- A ❌
  - 이는 정적 자격 증명을 사용하며, 비밀번호를 CloudFormation 템플릿(코드) 및 EC2 인스턴스 환경 변수/파일에 노출하게 되므로 보안 요구 사항을 위반

- B ❌
  - 이는 비밀번호를 코드에서 분리하고 보안을 향상시키는 좋은 방법이지만, 여전히 정적 사용자 이름과 비밀번호를 관리(생성, 순환)해야 함. C의 IAM 인증 방식이 비밀번호 자체를 없애므로 운영 노력이 더 적고 더 안전

- C ✅
  - `IAM 데이터베이스 인증`
    - Amazon Aurora(및 RDS)는 **IAM(Identity and Access Management)**을 사용하여 데이터베이스에 대한 액세스를 관리할 수 있음.
    - 이는 비밀번호를 사용하지 않고 단기 자격 증명을 생성하여 데이터베이스에 인증하는 방식 (실제로는 AWS API를 통해 발급된 임시 토큰을 비밀번호로 사용)
    - 보안 강화: 정기적으로 변경되는 임시 토큰을 사용하므로, 비밀번호 유출 위험이 없음
    - 최소 운영 노력: 비밀번호를 직접 관리, 순환, 저장할 필요가 없음. EC2 인스턴스에 연결된 IAM 역할만 관리하면 됨.

  - 방법
    - DB 클러스터 설정: Aurora에서 IAM 데이터베이스 인증을 활성화
    - DB 사용자 생성: 데이터베이스 내부에 IAM 사용자와 매핑되는 데이터베이스 사용자(예: `app_user`)를 생성
    - EC2 인스턴스 역할 연결: EC2 인스턴스에 IAM 역할을 연결. 이 IAM 역할에는 특정 Aurora DB 클러스터의 특정 DB 사용자(`app_user`)에 대해 `rds-db:connect`를 허용하는 정책이 있어야
    - 인증 흐름: EC2 인스턴스의 애플리케이션은 연결된 IAM 역할을 사용하여 AWS STS(Security Token Service)를 통해 임시 인증 토큰을 얻음. 이 토큰을 비밀번호 대신 사용하여 데이터베이스에 연결

- D ❌
  - IAM 사용자는 개별 사용자나 서비스 계정에 사용되지만, EC2 인스턴스 그룹과 같은 서비스에 자격 증명을 제공할 때는 IAM 역할을 사용하는 것이 표준적이고 더 안전한 방식
  - EC2 인스턴스와 IAM 사용자 자격 증명을 직접 연결하는 것은 보안적으로 권장되지 않으며 관리 복잡도가 높음. (IAM 역할을 사용하는 것이 EC2 인스턴스에 임시 자격 증명을 제공하는 가장 좋은 방법)
