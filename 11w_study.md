## 근거
A ❌
  - Amazon API Gateway와 AWS Lambda, 현대적인 아키텍처이지만, SNS 대신 SQS를 애플리케이션 서비스 간 통신 계층으로 사용하는 것은 적절하지 않음.
  - SQS는 주로 1:1 비동기 작업 처리에 사용되며, RESTful 서비스를 대체하는 데는 적합하지 않습니다. 또한 이 옵션은 확장 문제를 명확히 해결하지 못함.
B ❌
  - EC2 인스턴스 크기 증설,"**수직 확장(Scale Up)**은 임시방편일 뿐, 애플리케이션의 탄력성과 현대화 목표에 부합하지 않음.
  - 최대 사용률을 충족하더라도 갑작스러운 트래픽 증가나 예기치 않은 부하에는 여전히 취약하며, 비용 효율성이 떨어짐.
C ❌
  - Amazon SNS 사용, SNS는 게시/구독(Pub/Sub) 모델을 위한 서비스로, 주로 다수의 구독자에게 동일한 메시지를 브로드캐스트하는 데 사용.
  - 이는 메시징을 처리하고 단일 작업 부하를 분산하여 트랜잭션 손실을 방지해야 하는 이 시나리오(계층 간 1:1 작업 처리)에는 적합하지 않음. SQS가 작업 큐 역할을 하기에 올바른 선택.
D ✅
  - Amazon SQS 사용
    - 계층 간의 **동기적 통신(RESTful)**을 비동기 메시징으로 대체.
    - 트랜잭션이 삭제되는 문제를 근본적으로 해결. 요청이 즉시 처리되지 않아도 SQS 대기열에 안전하게 보관되므로 데이터 손실을 방지.
  - Auto Scaling 그룹의 EC2
    - 수요에 따라 자동으로 서버 인스턴스를 추가/제거하여 탄력적인 확장성을 제공하고 비용 효율성을 높임.
  - Amazon CloudWatch 및 SQS 대기열 길이 모니터링
    - SQS 대기열 길이는 백로그된 작업의 양을 나타내는 핵심 지표
    - 대기열 길이가 특정 임계값을 초과하면 과부하가 임박했음을 의미.
    - CloudWatch를 사용하여 이를 모니터링하고, 임계값을 초과하면 Auto Scaling 그룹이 새 EC2 인스턴스를 **자동으로 확장(Scale Out)**하도록 트리거하여 과부하를 즉시 해소.
  - 운영 효율성
    - SQS는 완전 관리형 서비스이며, CloudWatch 및 Auto Scaling을 통한 자동 확장은 관리 오버헤드를 최소화.


   
# 3번
## 정답
- C, D

## 근거
B ❌
  - 현재 문제의 근본적인 원인인 EC2 웹 서버를 통한 업로드를 유지하는 방법입니다. 이는 웹사이트 성능 개선 요구 사항을 충족할 수 없음.

C ✅
  - Presigned URL을 사용하면 사용자의 웹 브라우저가 EC2 인스턴스를 거치지 않고 Amazon S3로 직접 이미지를 업로드
  - 이 조치는 웹 서버(EC2)의 부하를 즉시 해소하고, 업로드 처리를 AWS의 S3 인프라로 오프로드하여 웹사이트의 응답 속도를 크게 개선
  - 결합 감소: 웹 서버의 역할을 순수하게 웹 요청 처리로 한정하고, 대용량 파일 업로드 처리를 S3에 위임하여 애플리케이션 계층 간의 책임을 분리 

D ✅
  - S3 이벤트 알림을 사용하여 이미지가 S3 버킷에 성공적으로 업로드되는 순간 (C 단계를 통해) AWS Lambda 함수를 자동으로 호출
  - Lambda 함수는 서버리스 방식으로 이미지 크기 조정 작업을 수행
  - 결합 감소: 웹 서버는 더 이상 크기 조정 작업을 담당하지 않으며, 이 작업을 완전 관리형 서버리스 서비스인 Lambda에 비동기적으로 위임합니다. 이는 **느슨한 결합(Loose Coupling)**을 실현
  - 운영 효율성: Lambda는 사용한 만큼만 비용을 지불하고, 서버 관리가 필요 없으므로 가장 운영 효율적인 방식 중 하나

## 정답
- D

## 근거
A ❌
  - IAM 정책은 개별 계정 또는 **개별 주체(사용자/역할)**에 적용.
  - 여러 계정에 대해 이 작업을 수동으로 반복해야 하므로 운영 오버헤드가 높음.
  - 또한, 계정 루트 사용자와 같은 일부 주체는 계속해서 인스턴스를 시작할 수 있는 권한을 가질 수 있어 완벽하게 통제하기 어려움.

B ❌
  - **AWS Resource Access Manager (RAM)**는 리소스를 다른 계정과 공유하기 위한 서비스.
  - 리소스 생성을 방지하는 거버넌스 도구가 아니므로 이 시나리오에 부적합.

C ❌
  - 옵션 A와 유사하게, 개별 계정마다 수동으로 IAM 정책/역할을 생성하고 관리해야 하므로 운영 오버헤드가 높음. 이는 중앙 집중식 거버넌스를 제공하지 않음.
  
D ✅
  - `AWS Organizations 사용`
    - AWS 계정을 통합적으로 관리하고 거버넌스를 중앙에서 적용하는 서비스
    - 여러 AWS 계정(개발, 스테이징, 프로덕션)에 걸쳐 단일 지점에서 정책을 적용
  - `서비스 제어 정책 (SCP) 사용`
    - Organizations 내의 모든 계정에 대해 최대 권한 경계를 설정하는 정책.
    - SCP는 IAM 사용자, 그룹, 역할의 권한을 재정의할 수 있으며, SCP에 의해 명시적으로 거부된 작업은 절대 허용되지 않음.
    - 모든 계정의 모든 주체(IAM 사용자/역할)가 특정 인스턴스 유형(대규모 인스턴스)을 시작하는 것을 확실하게 거부
  - 최소한의 운영 오버헤드
    - 관리 계정에서 SCP를 한 번 정의하고 모든 관련 계정에 적용하면, 이후 개별 계정에서 IAM 정책을 수동으로 업데이트하거나 관리할 필요가 없음.
    - 가장 운영 효율적인 방법으로 거버넌스 요구 사항을 충족

## 근거

A ❌
- Budgets를 설정할 때 태그를 필터링하여 부서별 예산을 만들 수는 있지만, **"Cost Explorer 예측 사용"**은 사전 경고 기능 때문에 Budgets와 함께 사용되는 더 나은 조합이며,
- D 옵션은 Budgets의 핵심 기능을 정확하게 설명하고 있츰.
- 태그 지정은 예산을 부서별로 나누는 데 필요할 뿐, 알림 자체를 구성하는 데 필요한 기능이 아님.

B ❌
  - `Cost Anomaly Detection`
  - **예상치 못한 급격한 비용 증가(이상 현상)**를 감지하는 데 사용됨.
  - 이는 예산의 특정 비율(60%)에 도달하는 예측 가능한 상황에 대한 알림을 설정하는 목적과는 다름.

C ❌
  - Trusted Advisor는 비용 최적화 권장 사항을 제공하며,
  - AWS Support API는 지원 티켓을 생성/관리하는 데 사용.
  - 이들은 예산 임계값 도달 알림을 구성하는 주된 도구가 아닙니다. Budgets가 이 목적을 위해 설계된 전용 서비스입니다.

D ✅
  - AWS Budgets
    - AWS에서 특정 금액 또는 사용량에 대한 예산을 설정하고, 실제 또는 예측 지출이 해당 예산을 초과하거나 임계값에 도달할 때 알림을 설정하는 데 사용되는 기본 서비스.
    - CFO가 예산의 **60%**에 도달했을 때 알림을 받기 원하는 요구 사항을 충족하는 데 가장 적합한 도구
  - AWS Cost Explorer
    - AWS Budgets는 지출 임계값뿐만 아니라 예측(Forecasted) 지출을 기반으로도 알림을 보낼 수 있음.
    - 이는 현재 지출 속도를 고려하여 예산 도달 시점을 미리 경고하여 사전 대응이 가능하게 함.
  - 60% 초과 시 알림 임계값 추가
    - AWS Budgets의 핵심 기능으로,
    - 사용자는 실제 지출 또는 예측 지출이 설정된 예산 금액의 **60%**와 같은 특정 비율에 도달할 때 알림(SNS 주제 또는 이메일)을 받도록 구성할 수 있음.
